# Bitmask 状态压缩与缓存优化原理解析

## 1. 为什么原有的 DFS 极慢？
原先在 `engine.py` 的 `DaVinciInferenceEngine` 中，DFS 递归穷举所有未知暗牌排列组合时，采用了大量的 Python 原生数据结构：
*   **列表分配与深拷贝**：例如对每张不确定的牌维护一个 `List[int]`，在每一层递归中需要用 `copy.deepcopy()` 去复制全场环境。
*   **字符串类型的字典键**：为了使用 Memoization (备忘录/缓存) 剪枝，原本必须将整个盘面序列化为一段极长的字符串或 JSON (比如 `json.dumps(slots)`) 作为 `dict` 的 Key。
*   **反复的 `in` 循环操作**：在过滤掉不可能的数字组合时，使用低效的 `[x for x in ... if x not in ...]` 循环。

在动辄需要验证几十万级树深度的达芬奇密码中，这成了压垮 CPU 和内存带栈的黑洞。

## 2. 位掩码 (Bitmask) 重构的核心思想
达芬奇密码的数字严格分布在 `{0, 1, 2, ..., 11}`（不考虑百搭 Joker 等特殊规则的情况下是 12 个槽），这极度契合 2 进制位运算优化！

我们**放弃用数组或者列表去表示“剩余可能性”，转而使用一个 16-bit 整数 (Int16)**：
*   如果一张黑牌可能是 0, 1, 3，我们将它的可能性集合压缩为：`(1 << 0) | (1 << 1) | (1 << 3) = 1 + 2 + 8 = 11 (0b00001011)`
*   如果确定一张牌就是 5，那么它的值就是 `(1 << 5) = 32 (0b00100000)`
*   如果我们已知对手手里绝不可能有黑色 7 `(1 << 7 = 128)`，我们要过滤他的暗牌，直接进行极其底层的位与非运算：`mask &= ~128`。速度直接拉满。

## 3. LRU 缓存 Key 的神级降维
通过将所有槽位的约束（12个槽的位图）收敛进一组 `Tuple[int]` 中：
*   旧的缓存 Key（几十个字节的长字符串与哈希计算）：`"{""0"":5,""1"":[2,3],...}"`
*   **新的缓存 Key（极其轻量的元组计算）**：`(5, 12, 1024, 0, ...)`
配合 `functools.lru_cache` 或直接在全局建立针对状态的 `memo`，可以将曾经要运行 20 分钟的十几个回归推演用例，压制在几秒钟甚至毫秒级内完成结算。响应移动端 App 的高频拖拽动作请求将毫无压力。
